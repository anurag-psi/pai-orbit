# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## What this repo is

`modo` is a Claude Code plugin ‚Äî a structured developer methodology harness distributed as a set of markdown command definitions, skill definitions, agent definitions, hook scripts, and templates. It has no build step and no test suite of its own.

The repo is installed into a target project by symlinking or copying `.claude-plugin/` into the target project's `.claude/plugins/modo/`. Claude Code then discovers commands, skills, agents, and hooks from that directory.

## Repository structure

```
.claude-plugin/
  plugin.json          ‚Äî plugin manifest (name, version, author)

commands/              ‚Äî slash commands that put Claude into a named mode
  build.md, design.md, domain.md, groom.md, plan.md, data.md, ux.md

skills/                ‚Äî slash commands that perform a defined operation
  board/, deploy/, git/, test/, review/, security-review/,
  incident/, analysis/, data-model/, simplify/, setup/, suggest-skills/
  (each has a SKILL.md)

agents/                ‚Äî sub-agents spawned by modes (primarily /build)
  docs-writer.md
  cross-repo-impact.md

hooks/                 ‚Äî shell scripts installed into a target project's .claude/hooks/
  bash-guard.sh        ‚Äî PreToolUse: blocks force-push, bulk staging, --no-verify, root rm -rf
  lint-python.sh       ‚Äî PostToolUse async: ruff check on edited .py files
  lint-ts.sh           ‚Äî PostToolUse async: eslint on edited .ts/.tsx files

templates/             ‚Äî files generated by /setup into the target project
  CLAUDE.md.template
  modo-config.md.template
  team.md.template
  agents/              ‚Äî stack-specific builder agent stubs (fastapi, nextjs, django, etc.)
  skills/              ‚Äî domain-operational skill template
  docs/                ‚Äî docs/ scaffold with starter files

docs/                  ‚Äî modo's own documentation (process-and-practices.md, capabilities.md, getting-started.md)
```

## Plugin architecture

Claude Code resolves plugin contents using the directory structure inside `.claude-plugin/`:

- **Commands** (`commands/*.md`) ‚Äî invoked with `/command-name`; the file content becomes Claude's system prompt for that mode
- **Skills** (`skills/<name>/SKILL.md`) ‚Äî invoked with `/skill-name`; the SKILL.md frontmatter `name` field is the trigger; content describes the procedure
- **Agents** (`agents/*.md`) ‚Äî spawned via Agent tool by modes; frontmatter `name`, `description`, `tools`, and `model` fields control how Claude Code surfaces them
- **Hooks** (`hooks/*.sh`) ‚Äî copied by `/setup` into the target project's `.claude/hooks/`; wired into `settings.json` at `hooks[].matcher` + `hooks[].command`

The plugin itself has no runtime ‚Äî it is purely declarative markdown read by Claude Code.

## Configuration model

Modes and skills are intentionally free of project-specific details. Project configuration lives in two files that `/setup` generates and modes read at runtime:

| File | Purpose |
|------|---------|
| `.claude/modo-config.md` | Board type and URLs, git branching model, deploy targets and auth checks, docs home |
| `.claude/team.md` | Team roster ‚Äî names, roles, handles ‚Äî used by `/board` for assignments |

Sections in `modo-config.md` are keyed by `## Heading` ‚Äî each skill reads only the section it needs (e.g., `/deploy` reads `## Deploy`, `/board` reads `## Agile Board`, `/git` reads `## Git`).

## Producer/consumer model

Every mode either produces docs or consumes them ‚Äî never both:

| Mode | Produces | Consumes |
|------|----------|----------|
| `/domain` | `docs/domain/` | ‚Äî |
| `/ux` | `docs/features/*/ux.md` | `docs/domain/` |
| `/groom` | `docs/features/*/requirements.md` | domain, ux |
| `/design` | `docs/features/*/design.md`, `docs/decisions/` | requirements, domain |
| `/build` | Code, `docs/domain/product-capabilities.md` | All docs, task board |
| `/test` | `docs/features/*/test-plan.md` | requirements |
| `/review` | `docs/wip/review-*.md` | diff, CLAUDE.md, decisions |
| `/plan` | `docs/plans/`, board moves | All docs, task board |
| `/data` | `docs/reports/` | database, domain |

`/plan` is the integrator ‚Äî it consumes everything and decides what to work on. Switch modes explicitly when headspace or output destination changes.

## Adding new content

**New mode** ‚Äî add `commands/<name>.md`. The file content is the full system prompt. Follow the established pattern: state what mode this is, when to switch out, what to read before starting, and what output to produce.

**New skill** ‚Äî add `skills/<name>/SKILL.md`. Frontmatter must include `name` (matches the slash command) and `description` (used by Claude Code to decide when to trigger it). Follow the board or deploy skill as a model.

**New agent** ‚Äî add `agents/<name>.md`. Frontmatter fields: `name`, `description`, `tools` (comma-separated list), `model`. The description is what Claude reads when deciding whether to spawn this agent.

**New hook** ‚Äî add `hooks/<name>.sh`. Update `templates/modo-config.md.template` if the hook needs project-specific configuration. Document it in `docs/capabilities.md`.

**New stack agent template** ‚Äî add `templates/agents/<stack>-builder.md`. Fill in `{{PLACEHOLDER}}` tokens that `/setup` replaces during generation.
